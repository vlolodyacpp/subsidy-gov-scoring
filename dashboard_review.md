# Оценка соответствия дашборда скоринг-системе

## Что дашборд отражает хорошо

1. **Все 12 метрик видны** — в детальном разборе заявки показываются все 12 факторов в виде горизонтального бар-чарта с вкладом каждого фактора в итоговый балл.

2. **Текстовая расшифровка** — группировка по 4 категориям (нормативное соответствие, сроки/бюджет, региональная специфика, характеристики заявки) с указанием процентного значения и вклада в баллах — это прямое отражение логики `scoring.py`.

3. **Распределение рисков** — donut-чарт с тремя уровнями (низкий ≥70, средний 45–70, высокий <45) точно соответствует порогам в скоринге.

4. **Фильтрация** — фильтры по региону, направлению, типу субсидии и диапазону баллов позволяют исследовать скоринг в разрезах, по которым он рассчитывается.

## Что дашборд отражает частично или не отражает

| Аспект | Проблема |
|--------|----------|
| **Веса метрик** | Нигде не показаны веса (10%, 12%, 8% и т.д.). Пользователь видит вклад в баллах, но не понимает, какой вес у метрики vs какое значение метрики |
| **Бюджетное давление** | Самый весомый фактор (12%), но нет отдельной визуализации динамики исчерпания бюджета по регионам/направлениям |
| **Очередь** | Queue position (10%) — нет визуализации хронологии подачи заявок и её влияния на баллы |
| **Нормативы** | Нет сравнения заявленного норматива с эталонным из справочника — пользователь не видит, откуда взялась оценка normative_match |
| **Сравнение заявок** | Нельзя сравнить две заявки бок о бок по 12 метрикам |
| **Региональная аналитика** | Показан только средний балл по регионам, но нет breakdown по факторам — непонятно, почему один регион выше другого |
| **Распределение по метрикам** | Гистограмма только по итоговому баллу. Нет гистограмм по отдельным факторам — нельзя увидеть, что, например, у большинства заявок плохой `amount_adequacy` |
| **Одиночная заявка через API** | При скоринге через `/score` budget_pressure, queue_position и unit_count получают дефолт 0.5 — дашборд это никак не предупреждает |

## Итоговая оценка: 7/10

Дашборд хорошо справляется с **обзорной аналитикой** и **детализацией отдельной заявки** — все 12 метрик доступны, есть текстовые объяснения и визуальный breakdown. Однако он работает скорее как **витрина результатов**, а не как инструмент для глубокого анализа механики скоринга. Не хватает:

- Визуализации весов и их влияния на итог
- Аналитики по отдельным метрикам (распределения, аномалии)
- Сравнительного анализа (заявка vs заявка, регион vs регион по факторам)
- Прозрачности для метрик с дефолтными значениями (0.5)

Если цель — прозрачность для комиссии, нужно добавить объяснение «почему именно этот балл», а не только «какой балл». Основа для этого уже есть в текстовом explanation, но визуально это недостаточно акцентировано.
