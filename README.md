# Subsidy Scoring System

**Кейс 2 — Скоринг сельхозпроизводителей при получении субсидий**
---
## Проблема

Сегодня субсидии на развитие животноводства в Казахстане распределяются по принципу **«первый подал — первый получил»**: финансирование уходит тому, кто раньше собрал документы, пока не закончился бюджет. Реальная эффективность хозяйства, история субсидирования и потенциал к росту — не учитываются.

Министерство сельского хозяйства хочет перейти к **merit-based скорингу** — распределению субсидий на основе данных о производителях.

## Решение

AI-система, которая **ранжирует заявки** по 11 факторам на основе Правил субсидирования (Приказ МСХ РК №108), **объясняет каждый score** (explainability) и **формирует shortlist** для рассмотрения комиссией. Финальное решение остаётся за человеком — система рекомендует, не решает.

### Что делает система

1. Загружает реестр заявок (34 584 записей после очистки) из xlsx
2. Заменяет нормативы из датасета на эталонные из справочника (Приложение 1 к Правилам)
3. Исключает отозванные заявки и заявки с нарушением дедлайна подачи из выборки
4. Извлекает 11 признаков из каждой заявки (feature engineering v2)
5. Рассчитывает score 0–100 через весовую формулу (11 факторов, 4 группы)
6. Генерирует текстовое объяснение: какие факторы повлияли и насколько
7. Формирует отсортированный shortlist с рекомендациями

## Данные

**Основной датасет:** реестр заявок на субсидирование за 2025 год, сформированный системой ИСС (subsidy.plem.kz). Получен 19.03.2026.

| Параметр | Значение |
|----------|----------|
| Записей (до очистки) | 36 651 |
| Записей (после очистки) | 34 584 |
| Одобренных | 31 675 |
| Отклонённых | 2 909 |
| Регионов | 19 |
| Направлений | 9 (скотоводство, овцеводство, коневодство и др.) |
| Типов субсидий | 47 |
| Суммы | от 300 до 1.23 млрд тенге |

### Нормативы

Нормативы субсидирования берутся **не из датасета**, а из справочника `src/normatives_dict.py`, составленного по **Приложению 1 к Правилам** (Приказ МСХ РК №108 от 15.03.2019, в ред. приказа от 18.11.2025 №428).

### Целевая переменная

Уточнённая бинарная целевая переменная:

| Класс | Статусы | Действие |
|-------|---------|----------|
| Positive (1) | Исполнена, Одобрена, Сформировано поручение, Получена | Включены |
| Negative (0) | Отклонена | Включены |
| Exclude | Отозвано | Исключены (инициатива заявителя, не решение комиссии) |
| Exclude | Нарушение дедлайна подачи | Исключены (по правилам сразу отклоняются) |

## Признаки (features) — 11 метрик

Из данных заявки извлекаются 11 числовых признаков, сгруппированных в 4 группы. Заявки с нарушением дедлайна подачи (вне окна 20 января — 20 декабря) исключаются на этапе загрузки данных, т.к. по правилам они сразу отклоняются.

### Группа 1: Нормативное соответствие (26%)

| # | Признак | Вес | Описание |
|---|---------|-----|----------|
| 1 | `normative_match` | 10% | Совпадение норматива заявки с эталонным из Приложения 1 |
| 2 | `amount_normative_integrity` | 8% | Арифметическая корректность: делимость суммы на норматив |
| 3 | `amount_adequacy` | 8% | Близость суммы к медиане для данного типа (z-score) |

### Группа 2: Сроки и бюджет (30%)

| # | Признак | Вес | Описание |
|---|---------|-----|----------|
| 4 | `budget_pressure` | 16% | Степень исчерпания бюджета по (регион, направление) на момент подачи |
| 5 | `queue_position` | 14% | Позиция в очереди по дате подачи внутри (регион, направление, тип) |

### Группа 3: Региональная специфика (29%)

| # | Признак | Вес | Описание |
|---|---------|-----|----------|
| 6 | `region_specialization` | 10% | Доля направления в общем числе заявок региона |
| 7 | `region_direction_approval_rate` | 12% | Одобряемость конкретного направления в конкретном регионе |
| 8 | `akimat_approval_rate` | 7% | Уровень одобрения акимата (МИО) |

### Группа 4: Характеристики заявки (15%)

| # | Признак | Вес | Описание |
|---|---------|-----|----------|
| 9 | `unit_count` | 5% | Количество заявленных единиц (перцентильный ранг внутри типа) |
| 10 | `direction_approval_rate` | 5% | Одобряемость направления |
| 11 | `subsidy_type_approval_rate` | 5% | Одобряемость типа субсидии |

## Модель скоринга

### Весовая формула (11 факторов)

Score = Σ (value_i × weight_i × 100), где value_i ∈ [0, 1], сумма весов = 1.0.

Результат:

| Метрика | Значение |
|---------|----------|
| Средний балл | 71.9 |
| Медиана | 72.3 |
| Диапазон | 33.6 — 96.4 |
| Ст. отклонение | 10.1 |

## Explainability

Каждый score сопровождается группированным объяснением. Пример вывода:

```
Score: 96.4/100 (Риск: низкий)

✓ Нормативное соответствие: сильная (24.3/26 баллов)
    Соответствие норматива эталону: 100% → +10.0
    Арифметическая корректность суммы: 100% → +8.0
    Адекватность суммы заявки: 79% → +6.3

✓ Сроки и бюджет: сильная (29.8/30 баллов)
    Бюджетное давление: 100% → +16.0
    Позиция в очереди подачи: 98% → +13.7

✓ Региональная специфика: сильная (28.5/29 баллов)
    Профильность направления для региона: 95% → +9.5
    Одобряемость направления в регионе: 100% → +12.0
    Уровень одобрения акимата: 100% → +7.0

✓ Характеристики заявки: сильная (13.8/15 баллов)
    Количество заявленных единиц: 85% → +4.2
    Одобряемость направления: 92% → +4.6
    Одобряемость типа субсидии: 99% → +5.0
```

В будущем данная система будет дополнена работой с ML-моделью, которая будет использоваться для скоринга заявок на основе рассчитанных признаков.

## Быстрый старт

### Вариант 1: Docker Compose

```bash

# 2. Поместить данные в data/subsidies.xlsx
```

### Пример запроса POST `/score`

```json
{
  "region": "СКО",
  "direction": "Субсидирование в мясном и мясо-молочном скотоводстве",
  "subsidy_type": "Заявка на получение субсидий за приобретение племенных быков-производителей мясных и мясо-молочных пород",
  "district": "г. Петропавловск",
  "akimat": "Акимат СКО",

  "submit_day": 15

```

Поле `normative` не передаётся — система автоматически определяет его из справочника по `subsidy_type`.

## Сценарии использования

**Аналитик МСХ формирует shortlist:** загружает заявки → система рассчитывает scores по 11 факторам → отсортированный shortlist → передаёт комиссии.

**Член комиссии проверяет заявку:** выбирает заявку в дашборде → видит score + разбивку по группам факторов → понимает обоснование → принимает решение.

**Региональное управление анализирует:** видит распределение scores по региону → выявляет слабые направления → корректирует работу.
