# Models

Эта папка хранит артефакты ML-модели для проекта скоринга субсидий.

## Что будет лежать здесь

- `artifacts/` — сохранённые модели: `model.joblib`, `model.pkl`, `model.onnx`
- `reports/` — метрики обучения, feature importance, confusion matrix, ROC-AUC

## Как устроена модель

Текущий проект уже умеет извлекать хорошие табличные признаки:

- нормативное соответствие
- адекватность суммы
- арифметическую корректность суммы
- типичность суммы для выбранного типа субсидии
- типичность масштаба заявки без использования approval-history

Просроченный дедлайн теперь не участвует в score и не подаётся в ML:
такие заявки автоматически отсекаются как `eligibility-fail`.

Поэтому ML-слой строится не "с нуля по сырому Excel", а поверх этих признаков.

Текущий production-вариант:

1. Сначала строим `features_df` из `src/features.py`
2. Используем historical outcome label как supervision target
3. Веса объектов корректируем через `merit_proxy_score`, собранный из внутренних признаков заявки
4. Обучаем `dual-branch ensemble`:
   - `context branch`: `RandomForest`, видит ограниченный контекст заявки, включая `region`
   - `core branch`: `LogisticRegression`, получает region-neutral view и не может опираться на `region` напрямую
5. Смешиваем вероятности двух веток на validation
6. Переводим итоговую вероятность в `ml_score` от 0 до 100

## Как модель будет работать

Продакшн-поток можно сделать таким:

1. Новая заявка проходит через текущий pipeline
2. Если дедлайн просрочен, заявка сразу отбраковывается и не попадает в ML
3. Для допустимой заявки извлекаются признаки
4. Загружается сохранённая модель из `artifacts/`
5. Dual-branch модель считает итоговую силу заявки
6. Исторический approval-history подключается отдельно как advisory, а не как runtime-признак primary model

Текущая production-логика:

`final_score = ml_score`

Такой подход хорош тем, что:

- сохраняется прозрачность текущего rule-based механизма как диагностического слоя
- появляется адаптация к реальным историческим данным
- проще объяснять результат комиссии
- итог остаётся предварительной рекомендацией, а не юридически значимым решением

## Как обучать модель

Базовый цикл обучения:

1. Запустить `run_pipeline()` и получить очищенный датафрейм
2. Построить таблицы через `build_feature_tables(df)`
3. Получить признаки через `extract_features_batch(df, tables)`
4. Сформировать `X = primary_model_frame`
5. Сформировать `y = historical is_approved`
6. Отдельно посчитать `merit_proxy_score` и использовать его для sample weighting и диагностик
7. Разделить данные на train/valid/test
8. Обучить обе ветки модели
9. Подобрать веса веток, `temperature scaling` и `decision threshold`
10. Проверить метрики: `ROC-AUC`, `F1`, `precision`, `recall`
11. Сохранить итоговый bundle в `artifacts/`

## Что уже готово в коде

В репозитории теперь есть:

- `src/modeling.py` — общие функции для подготовки датасета, split, обучения и сохранения модели
- `train.py` — обучение модели и сохранение артефактов
- `predict.py` — batch-инференс по обученной модели

Текущее обучение использует:

- `RandomForest` как context branch
- `LogisticRegression` как core branch
- подбор веса веток на validation

## Команды запуска

Установка зависимостей:

```bash
python3 -m venv .venv
.venv/bin/python -m pip install --upgrade pip
.venv/bin/python -m pip install -r requirements.txt
```

Обучение модели:

```bash
.venv/bin/python train.py
```

После обучения появятся файлы:

- `models/artifacts/subsidy_model.joblib`
- `models/reports/training_metrics.json`
- `models/reports/test_predictions.csv`

Запуск предикта:

```bash
.venv/bin/python predict.py
```

Результат batch-предикта сохранится в:

- `models/reports/predictions.csv`

## Интеграция с API

Если файл `models/artifacts/subsidy_model.joblib` существует, то `src/api.py`
автоматически загрузит его при старте сервера и переключится на merit scoring:

- `score` станет итоговым dual-branch merit score
- `rule_score` останется как диагностический rule-based score
- `ml_score` и `ml_probability` будут добавлены в ответы API
- history advisory будет возвращаться отдельно и не будет напрямую управлять итоговым score

API ожидает, что модель уже обучена.
Если файла модели нет, сервер не стартует и попросит сначала выполнить `.venv/bin/python train.py`.

## Что важно для качества

- Просроченный дедлайн теперь означает мгновенную дисквалификацию:
  итоговый результат принудительно считается отрицательным независимо от ML-сигнала
- criteria из приложения 2 покрыты только частично:
  проверки, требующие ГИСС, ИСЖ, ИБСПР, ЕАСУ, ИСЕГКН и документов, остаются на manual review
- Делать разбиение по времени, если хотите честно проверять качество на "будущих" заявках
- Следить за дисбалансом классов
- Не использовать признаки, которые знают будущее и приводят к leakage
- Не давать `region` полностью доминировать: одна ветка должна уметь работать в region-neutral режиме
- Сохранять версию модели и список признаков рядом с файлом модели

Текущая реализация уже уменьшает leakage:

- approval-rate признаки считаются по прошлой истории, а не по всему датасету сразу
- `queue_position` больше не использует будущее число заявок в группе
- `budget_pressure` больше не смотрит на итоговые будущие approved totals
- просроченные заявки вообще не попадают в train/valid/test
- в `training_metrics.json` сохраняются диапазоны дат по `train/valid/test`
